# 🚀 Antigravity Agentic Coding Rules & Guidelines
> 본 문서는 Antigravity AI 코딩 어시스턴트가 사용자(USER)와 협업하며 상시 준수해야 하는 **글로벌 기본 규칙 및 세부 명령 규칙**을 정의한 문서입니다.
> 어떠한 예외 상황에서도 이 규칙은 최우선순위로 엄격하게 강제 적용됩니다.

---

## 🎨 1. 언어 및 주석 규정 (Language & Comments)

* **의사소통 언어**: 대화, 계획서, 상태 보고 등 모든 답변은 **100% 한국어**로 진행합니다.
* **코드 주석**: 코드를 새로 작성하거나 수정할 때, 소스코드 내부의 모든 설명 주석은 반드시 **한글**로 명확히 작성하여 추가해야 합니다.

---

## 🔒 2. 환경변수 및 보안 규정 (Security & Environment Variables)

> [!CAUTION]
> **`.env` 파일은 보안상 매우 민감한 파일이므로 절대 읽거나 직접 수정해서는 안 됩니다.**

* **대체제 활용**: 설정 값이나 환경 변수 변경 및 추가가 필요할 경우, 절대 `.env`를 만지지 말고 `.env-sample` 파일을 생성하거나 수정하여 명세서 가이드라인을 보완해야 합니다.
* **유출 방지**: `.env`에 지정된 실서버 자격 증명이 실수로도 Git에 추적되지 않도록 철저히 차단합니다.

---

## 📋 3. 작업 계획 및 승인 프로세스 (Process & Approval)

> [!IMPORTANT]
> **모든 작업은 반드시 계획을 먼저 공유하고 사용자 승인 후에 수행합니다.**

* **계획 의무화**: 사용자의 요청을 수행할 때 항상 구체적인 실행 계획을 수립해야 합니다.
* **파일 계획서 작성**: 단순 설명이 아닌, 계획 단계에서 `implementation_plan.md` 등의 계획 파일을 생성하여 사용자에게 시각적으로 제안해야 합니다.
* **삭제 확인 필수**: 기존 파일 및 디렉토리 삭제(`rm` 등) 작업은 어떠한 경우라도 **사전에 삭제 허가를 묻고 승인받은 후** 진행해야 합니다.

---

## 💻 4. 터미널 명령어 실행 정책 (Terminal Commands Policy)

터미널 명령어를 실행할 때 승인 여부와 효율적인 입력을 위해 다음과 같은 기준을 따릅니다.

### 🟢 자동 실행 허용 (Safe to AutoRun)
아래와 같이 **파일을 수정하거나 삭제하지 않는 단순 조회 및 테스트 명령어**는 사용자 승인 없이 즉시 실행이 가능합니다.
* **단순 파일 조회**: `ls`, `find`, `grep` 등
* **빌드 및 테스트**: `xcodebuild`, `npm run build` 등
* **네트워크 상태 확인**: `ping`, `curl` (단순 진단 목적) 등

### 🔴 사전 승인 필수 (Requires Approval)
파일을 수정 및 삭제하거나 프로젝트 형상관리에 영구적인 영향을 끼치는 명령어는 반드시 **사전 계획 승인 후**에 실행해야 합니다.
* **파일 삭제 및 파괴**: `rm`, `git reset`, `clean` 등
* **코드 수정 및 배포**: 소스코드 내용 변경 등
* **Git 이력 기록 및 원격 전송**: `git commit`, `git push` 등

### ⌨️ 명령어 포맷 규칙
* 명령어 실행 시 사용자가 터미널에서 여러 번 엔터를 치는 번거로움이 없도록, 가급적 **줄바꿈(`\`) 없이 한 줄로 합쳐서 실행**하거나 가장 깔끔하고 간결한 포맷을 유지합니다.

## 🐙 5. Git 협업 및 형상 관리 (Git Management)

* **Git 로컬 커밋 자동화**: 작업 단위나 구현이 완료되면 **로컬 Git 커밋(`git commit`)은 AI가 자동으로 즉시 실행**하여 안전하게 코드를 백업합니다.
* **한글 커밋 코맨트**: 로컬 커밋 시 작성하는 커밋 메시지(코맨트)는 반드시 **한글**로 품격 있고 명확하게 작성합니다.
* **원격 Push 제한**: 원격 저장소로의 업로드(`git push`)는 절대로 독단적으로 진행하지 않으며, **사용자가 명시적으로 푸시를 요청할 때만** 실행합니다.

## 📝 6. 작업 이력 로깅 (Work Logging)

> [!NOTE]
> **마지막 작업 내용은 항상 문서로 기록하여 유산으로 유지 관리해야 합니다.**

* **로그 작성 규칙**: 매 작업이 마무리될 때마다 루트 디렉토리 아래의 `/doc/마지막작업-오늘날짜.md` 파일로 작성하여 추가합니다.
* **기존 내용 유지**: 기존 기록을 훼손하거나 덮어쓰지 않고, **기존 내용을 안전하게 유지하면서 하단에 새로운 작업 내용을 추가**하는 방식으로 문서를 관리합니다.
